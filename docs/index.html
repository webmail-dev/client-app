Angular 21 + PWA + plantilla mobile delivery

Documentación técnica de Dale Pues

Dale Pues es una aplicación Angular mobile-first que migra cinco vistas HTML estáticas de una plantilla de delivery a una PWA modular con rutas, componentes compartidos, carga controlada de scripts visuales y assets cacheables.

Arquitectura

La aplicación es un frontend Angular standalone. No contiene backend ni clientes HTTP de datos; su comportamiento actual se concentra en navegación SPA, composición visual, PWA y adaptación de plugins heredados de una plantilla HTML. 

Navegador / PWA
  |
  |-- Angular bootstrapApplication(App, appConfig)
  |   |-- Router lazy-loaded por página
  |   |-- Componentes compartidos mobile-first
  |   |-- Servicios de integración visual
  |   `-- Service worker Angular en producción
  |
  |-- Assets estáticos
  |   |-- CSS: Metropolis, Remixicon, Swiper, Bootstrap, tema Zomo
  |   |-- JS: Bootstrap global, Swiper diferido
  |   `-- Imágenes por dominio: landing, food, grocery, pharmacy, profile
  |
  `-- Navegación SPA
      |-- /
      |-- /food
      |-- /grocery
      |-- /pharmacy
      `-- /profile
La arquitectura conserva la estructura visual de los HTML fuente, pero separa layout, páginas y scripts en piezas Angular mantenibles.

Flujo de navegación

El flujo principal se realiza desde BottomNav. La página raíz permite navegar a las verticales mediante routerLink en las tarjetas Food, Grocery y Pharmacy. Las acciones internas de catálogo todavía apuntan a #, porque la migración cubre únicamente las cinco vistas solicitadas.

PWA

Manifest

public/manifest.webmanifest define name y short_name como Dale Pues, color de tema #dc3545, modo standalone, scope raíz, start_url raíz e iconos de 192 y 512 px.

Service Worker

app.config.ts registra ngsw-worker.js con provideServiceWorker. El service worker solo se habilita fuera de modo desarrollo mediante !isDevMode().

GrupoEstrategiaRecursosObjetivo
appprefetchindex.html, manifest, favicon, CSS y JS raíz.Cachear shell de aplicación al instalar.
assetslazy + prefetch en updates/assets/**, /icons/**Cachear imágenes, fuentes y recursos estáticos cuando se usan.
navigationUrlsSPA fallback/** excluyendo archivos con extensión.Permitir navegación directa a rutas Angular.
El soporte offline cubre el shell Angular y assets cacheados. No hay sincronización offline de datos porque la app actual no consume APIs de negocio.

Autenticación PocketBase

La primera fase de autenticación usa PocketBase como backend de identidad y mantiene los datos de perfil en la colección Auth users. La arquitectura queda preparada para separar comercios y repartidores en merchant_profiles y courier_profiles cuando avance el dominio.

Servicios Angular PocketbaseService y AuthPocketbaseService

Exponen la instancia SDK, login con email/password, Google OAuth2, registro, logout, recuperación de contraseña, estado de sesión, roles y verificación de perfil incompleto.

Rutas nuevas /login, /register, /complete-profile, /home

Las vistas principales están protegidas por authGuard. El formulario de completar perfil bloquea el acceso si faltan datos mínimos o términos aceptados.

Esquema de usuarios

La migración backend/pb_migrations/20260624000100_update_users_auth.js agrega campos operativos a users: phone, type, status, address, city, state, country, roleDescription, lastLoginAt, profileCompleted, termsAccepted y campos temporales mínimos para comercio/repartidor.

CampoValores / uso
typeclient, merchant, courier, admin, support.
statusactive, pending, rejected, blocked, inactive.
profileCompletedCalculado por Angular según los campos requeridos para cada tipo de usuario.
businessName, businessTypeCampos temporales de fase 1 para comercios. TODO: mover a merchant_profiles.
identityDocument, vehicleTypeCampos temporales de fase 1 para repartidores. TODO: mover a courier_profiles.

Flujo

En login tradicional se llama a authWithPassword. En Google se llama a authWithOAuth2({ provider: 'google' }). Después de autenticar, el servicio normaliza type, status, country, actualiza lastLoginAt y redirige a /complete-profile si faltan datos. Las cuentas blocked o inactive limpian sesión y no acceden al panel.

Operación PocketBase

cd backend
./pocketbase migrate up --dir pb_migrations
./pocketbase serve

Para Google OAuth2, habilitar el proveedor en PocketBase Admin dentro de la colección users, registrar Client ID y Client Secret de Google Cloud, y usar como Authorized redirect URI la URL OAuth mostrada por PocketBase para el servidor activo.

Troubleshooting

ProblemaCausa probableSolución
Assets no encontradosLa app usa assets/... y Angular copia public/**.Verificar que el archivo exista en public/assets. Evitar depender de src/assets mientras no esté configurado.
CSS duplicado o estilos inconsistentesLos CSS también están enlazados en src/index.html y cargados desde angular.json.Centralizar la carga en una sola estrategia si aparecen duplicaciones visuales.
Sliders no inicializan al cambiar de rutaSwiper necesita inicializarse después de renderizar la vista.Usar VisualPluginService.initPagePlugins(true) en AfterViewInit.
